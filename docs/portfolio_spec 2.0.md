# Portfolio Specification: Telegram-бот доставки еды "NeoPizza Dark Kitchen"

**Версия:** 2.0  
**Статус:** ✅ Финализирован  
**Назначение:** Portfolio + GitHub

---

## 1. Executive Summary

Продукт — D2C (Direct-to-Consumer) платформа доставки еды на базе Telegram. Формат: **100% Dark Kitchen с опцией самовывоза**. Бот является единым интерфейсом для клиентов и лёгкой ERP-системой для персонала.

Интеграция с локальной кассой (iiko / r_keeper) намеренно вынесена за рамки v1: кассир вручную переносит данные заказа в стационарную систему. Бухгалтер получает выгрузку напрямую из бота — список ингредиентов, расчёт по технологическим картам и финансовые отчёты.

**Ключевые бизнес-цели:**
- Уйти от 20–35% комиссии агрегаторов.
- Обеспечить чекаут за 3 клика — быстрее, чем у конкурентов.
- Снизить зависимость от минимального чека за счёт самовывоза.
- Защитить пользовательские данные в соответствии с ФЗ-152 / GDPR через hard gate при первом запуске.

## 2. Business Direction Analysis

Среди рассмотренных вариантов (шаурмичная, столовая, кафе, ресторан) выбрана **Сетевая Пиццерия**.

**Обоснование:**
- Пицца идеально выдерживает транспортировку, Food Cost ~20–25% — высокая маржинальность.
- Легко преодолевает минимальный порог заказа (1 200 ₽) даже на одного-двух человек при доставке.
- Самовывоз полностью снимает барьер минимального чека для одиночных клиентов — конверсия растёт.
- Операционная модель пиццерии (конвейер: приём → кухня → выдача/доставка) линейно ложится в FSM-архитектуру бота.

**Формат:** Dark Kitchen + Самовывоз.  
**Позиционирование:** Супер-быстрая технологичная пицца без приложений и звонков. Целевая аудитория — digital-native: геймеры, айтишники, офисные работники. Минималистичный текстово-визуальный интерфейс, акцент на скорость и предсказуемость. Заказ — без отрыва от мессенджера.

## 5. Ключевые продуктовые механики

### 5.1 Privacy Policy — Hard Gate

При первом запуске (`/start`) бот **полностью блокирует** любое взаимодействие до нажатия кнопки **[Принимаю условия обработки ПД]**. Без согласия — ни меню, ни заказы недоступны. Соответствие: ФЗ-152 (Россия) / GDPR (ЕС).

### 5.2 Самовывоз

Клиент выбирает режим в самом начале флоу: **Доставка** или **Самовывоз**.

- При самовывозе: минимальный порог заказа (1 200 ₽) **не применяется**.
- Клиент выбирает ближайшую точку и удобное время получения.
- Стоимость доставки = 0. Вся логистика отсутствует.
- Курьер в этом флоу не задействован.

### 5.3 Допродажа до минимального чека

При доставке, если сумма корзины < 1 200 ₽, бот **не блокирует** переход жёстко, а мягко предлагает добавить позиции:

> *«До минимального заказа осталось 340 ₽. Добавить напиток или соус?»*

Показывается карусель популярных дополнений. Только если клиент отказывается — блокировка с предложением переключиться на самовывоз.

### 5.4 Динамический ETA

ETA рассчитывается по формуле при каждом новом заказе:

```
ETA = (Активные_заказы × Среднее_время_приготовления) + Время_логистики(геозона)
```

При ETA выше порогового значения (например, > 60 минут) бот показывает **алерт**:

> *«Сейчас высокая нагрузка. Ожидаемое время доставки: 75 мин. Выбрать самовывоз (~15 мин)?»*

Клиент может согласиться на ожидание или переключиться на самовывоз — без потери корзины.

### 5.5 Геозоны доставки

Каждая точка имеет полигон зоны доставки (GeoJSON). Стоимость и ETA рассчитываются по тарифной сетке:

| Зона | Расстояние | Стоимость | Доп. время |
|------|-----------|-----------|------------|
| A | 0–2 км | 99 ₽ | +15 мин |
| B | 2–4 км | 179 ₽ | +25 мин |
| C | 4–7 км | 269 ₽ | +40 мин |

Адрес за пределами зоны — вежливый отказ с предложением самовывоза.  
Расчёт расстояния: формула Haversine или OSRM (матрица маршрутов).

### 5.6 Биржа заказов — назначение курьера

Двухэтапная стратегия через Celery:

1. Готовый заказ публикуется в **общий пул**: все свободные курьеры точки получают уведомление с кнопкой «Взять заказ». Достаётся первому.
2. Если за **3 минуты** никто не откликнулся → Celery-задача принудительно назначает курьера, закреплённого за геозоной точки.
3. Если районный курьер недоступен → уведомление администратору.

### 5.7 Курьерский фрод — только ручной ревью

Если курьер фиксирует отказ клиента от оплаты при доставке:
- Курьер нажимает «Клиент не заплатил» → жалоба автоматически отправляется в чат **Админа**.
- **Никаких автоматических банов.** Только ручной ревью администратором.
- Админ изучает ситуацию и принимает решение: предупреждение, ограничение способа оплаты или полная блокировка.
- История инцидентов хранится в системе для анализа паттернов.

### 5.8 Возврат средств

При отмене онлайн-оплаченного заказа кассиром — возврат через Telegram Payments API **автоматически**, без ручного вмешательства. Клиент получает уведомление с причиной отмены и подтверждением возврата.

## 6. Full Roadmap

### Phase 0 — Infrastructure & Legal Foundation
- Развёртывание Docker-инфраструктуры (PostgreSQL, Redis, Celery, Nginx).
- CI/CD pipeline: GitHub Actions (lint, typecheck, test, deploy).
- Экран Privacy Policy — hard gate при `/start`.
- Базовая ролевая система (Middleware + JWT/session).

### Phase 1 — Client Flow (Core)
- Выбор режима: **Доставка / Самовывоз**.
- Каталог с фото, категории, корзина (Redis).
- Геолокация + проверка геозоны + расчёт стоимости доставки.
- Допродажа до 1 200 ₽ (мягкий upsell).
- Выбор точки и времени для самовывоза.
- Оформление заказа и подтверждение с ETA.

### Phase 2 — Payments & Order Lifecycle
- Интеграция Telegram Payments API (онлайн-оплата).
- Оплата при доставке (наличные / карта курьеру).
- Автоматический возврат при отмене.
- Трекинг статусов: `🟡 Принят → 🔵 Готовится → 🟠 В пути → 🟢 Доставлен`.
- Уведомления клиенту при каждой смене статуса.

### Phase 3 — Staff Operations
- Интерфейс Кассира: подтверждение/отклонение, список смены.
- Интерфейс Шефа: очередь заказов, статус «Готово».
- Интерфейс Курьера: биржа заказов, маршрут, статус доставки, фиксация оплаты, жалоба на фрод.
- Интерфейс Бухгалтера: выгрузка чеков, ингредиентов по технологическим картам, финансовые отчёты по сменам.
- Динамический ETA + алерт пиковой нагрузки.

### Phase 4 — Admin Panel & Dynamic ETA
- Команда `/admin`: управление меню, персоналом, промокодами, расписанием.
- Команда `/superadmin`: аналитика по всем точкам, управление ролями, ручной ревью инцидентов.
- Алгоритм динамического ETA с учётом текущей загрузки кухни.
- Настройка геозон и тарифной сетки доставки через админку.

### Phase 5 — Polish & Portfolio Launch
- Unit-тесты сервисного слоя (pytest, покрытие ≥ 60%).
- README с архитектурной схемой (Mermaid), инструкцией запуска, скриншотами.
- ARCHITECTURE.md — описание FSM, слоёв, паттернов, алгоритмов ETA и геозон.
- Нагрузочное тестирование. Публикация на GitHub.

## 7. Financial Model Structure (Юнит-экономика)

**Ключевые показатели на 1 заказ (AOV ~ 1 500 ₽):**

| Статья | Доставка | Самовывоз |
|--------|----------|-----------|
| Выручка | 1 500 ₽ | 1 500 ₽ |
| COGS (продукты, ~22%) | −330 ₽ | −330 ₽ |
| Эквайринг (~2.5%) | −37 ₽ | −37 ₽ |
| Логистика (курьер, ср.) | −250 ₽ | 0 ₽ |
| **Contribution Margin** | **~59%** | **~76%** |

**Стратегические выводы:**
- Доля самовывоза 25% существенно улучшает общую маржинальность портфеля.
- Алерт пиковой нагрузки с переключением на самовывоз снижает потери лояльности и расходы на возвраты при перегрузах.
- Мягкая допродажа до 1 200 ₽ увеличивает AOV без раздражения клиента.

## 9. Risk Register

| Риск | Влияние | Стратегия смягчения |
|------|---------|---------------------|
| **Пиковая нагрузка** | High | Динамический ETA + обязательный алерт с предложением самовывоза. Клиент делает осознанный выбор — нет неожиданных задержек и возвратов. |
| **Курьерский фрод (ложный невыкуп)** | Medium | Только ручной ревью Админом. Автоматических банов нет — исключает несправедливые блокировки, сохраняет доверие курьеров. |
| **Юридический (ПД / ФЗ-152 / GDPR)** | Critical | Hard gate: бот недоступен до нажатия [Принимаю ПД]. Данные хранятся на серверах в РФ. Compliance — в приоритете перед запуском. |
| **Недостижение минимального чека** | Low | Мягкая допродажа напитков/соусов. Переключение на самовывоз без лимита. Жёсткая блокировка — только крайний fallback. |
| **Двойное бронирование заказа** | Medium | Атомарные транзакции БД + Redis lock при создании заказа. |
| **Утечка ПД (PII)** | Critical | Шифрование телефонов и адресов в БД. Автоудаление данных через 3 года. Логи без PII. |

## 10. Assumptions (Зафиксированные допущения)

1. **Касса:** Кассир вручную переносит данные заказа в стационарную систему (iiko / r_keeper). Интеграция по API — за рамками v1.
2. **Бухгалтер:** Расчёт ингредиентов и костов генерируется ботом на основе технологических карт блюд, зашитых в БД администратором.
3. **Геозоны:** Расчёт дистанции через формулу Haversine (v1). Опционально — OSRM / Yandex Matrix API для точного маршрута (v2).
4. **Курьерский фрод:** Никаких автоматических блокировок клиентов. Все инциденты — через ручной ревью Супер-Админа.
5. **Самовывоз:** Не требует курьера и логистики. Отдельный FSM-флоу с выбором точки и времени получения.
6. **Предзаказ:** Клиент может оформить заказ вне рабочих часов точки с выбором времени открытия. Celery Beat активирует его автоматически.
